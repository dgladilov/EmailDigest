# Рассылка о релизе фичей

Контент вы пишете только в `content.json`. Вёрстку (`template.html`) трогать
не нужно. Запускаете генератор — получаете готовый `email.html` для отправки.

---

## Быстрый старт

1. Установите Node.js (если ещё нет): https://nodejs.org — подойдёт любая LTS-версия.
2. Один раз выполните `npm install` в этой папке (ставит nodemailer для отправки).
3. Откройте `content.json` в любом редакторе и заполните под свой релиз.
4. Соберите письмо:

       node build.js

5. Проверьте `email.html` в браузере, затем отправьте:

       node send.js --to team@company.ru

   (настройки SMTP — см. раздел «Отправка через SMTP» ниже)

Можно указать свои пути:

    node build.js my-release.json out.html

---

## Структура content.json

Файл — это один объект. Сверху общие поля письма, затем массив `features`
с фичами, и в конце — опциональные блоки. Пример минимального файла:

    {
      "release_title": "Релиз июня",
      "intro": "Что вошло в этот релиз — ниже.",
      "config": { "image_mode": "inline", "max_inline_image_kb": 1536 },
      "features": [ ... ],

      "resources": [ ... ],
      "contact": "@dmitrii в Slack",
      "team": "Команда мобильной разработки"
    }

Обязательны только `release_title`, `intro` и `features`. Объект `config`
и всё, что после `features` (`resources`, `contact`, `team`), можно опустить.

### Общие поля письма

| Поле              | Обязательное | Что это                                          |
|-------------------|:------------:|--------------------------------------------------|
| `release_title`   |      да      | Заголовок в зелёной шапке письма                 |
| `intro`           |      да      | Вводный абзац под шапкой                          |
| `config`          |     нет      | Объект настроек письма (см. ниже)                 |
| `features`        |      да      | Массив фичей (см. ниже)                           |
| `resources`       |     нет      | Блок «Полезные материалы» — ссылки после фичей    |
| `resources_title` |     нет      | Заголовок блока материалов (по умолч. «Полезные материалы») |
| `contact`         |     нет      | Куда писать вопросы (в подвале)                   |
| `team`            |     нет      | Подпись команды в подвале                         |

### Объект config

Настройки сборки письма собраны в отдельный объект `config`:

    "config": {
      "image_mode": "inline",
      "max_inline_image_kb": 1536
    }

| Поле                  | По умолчанию | Что это                                    |
|-----------------------|:------------:|--------------------------------------------|
| `image_mode`          |  `"inline"`  | Как подключать картинки: `"inline"` (вшиты в HTML) или `"cid"` (вложения) |
| `max_inline_image_kb` |  `1536`      | Порог размера картинки в КБ (1536 = 1.5 МБ) |

Объект целиком опционален: нет `config` — действуют значения по умолчанию.
Можно указать только одно из полей.

Блоки после списка фичей — опциональные:

- Нет `resources` (или пустой массив) → блок «Полезные материалы» не показывается.
- Нет `contact` и `team` → подвал не показывается вообще.
- Есть только одно из `contact`/`team` → показывается только оно.

### Полезные материалы

`resources` — массив ссылок, которые показываются отдельным блоком после
всех фичей. Каждый элемент:

    { "title": "Полная таблица релиза", "url": "https://confluence.example.com/..." }

- `url` — обязателен (без него ссылка пропускается).
- `title` — текст ссылки. Если не задан, показывается сам URL.
- Порядок и количество — любые.
- Заголовок блока можно поменять полем `resources_title` на верхнем уровне.

Пример:

    "resources": [
      { "title": "Полная таблица релиза (Confluence)", "url": "https://confluence.example.com/release-june" },
      { "title": "Дашборд метрик", "url": "https://grafana.example.com/dashboard" },
      { "title": "Чек-лист регресса", "url": "https://example.com/regress-checklist" }
    ]

## Как описывать фичу

Каждая фича — объект внутри массива `features`. Поля:

| Поле            | Обязательное | Что это                                              |
|-----------------|:------------:|------------------------------------------------------|
| `name`          |      да      | Название фичи                                        |
| `description`   |     нет      | Краткое описание. Нет поля — строка не показывается   |
| `analytics_url` |     нет      | Ссылка на аналитику. Нет — ссылка не показывается     |
| `mockup_image`  |     нет      | Путь к локальному файлу макета — вшивается в письмо   |
| `mockup_url`    |     нет      | Ссылка на макет (полный/фолбэк для большого файла)    |
| `platforms`     |     нет      | Список платформ с версиями (бейджи над названием)     |

### Опциональные поля — как это работает

Все поля кроме `name` можно не указывать. Правила отображения:

- Нет `analytics_url` → ссылка «Аналитика» не показывается.
- Нет `description` → строка описания не выводится.
- Нет `platforms` (или пустой массив) → бейджи не выводятся.
- Разделитель «|» между ссылками появляется только когда рядом есть две ссылки.

Достаточно просто не писать поле в JSON (либо оставить пустую строку `""`).

### Макеты: картинка или ссылка

У макета два поля, работают вместе:

- `mockup_image` — путь к картинке на диске (относительно `content.json`).
  Если файл найден и не превышает порог размера — картинка вшивается прямо
  в письмо. Форматы: png, jpg, gif, webp.
- `mockup_url` — ссылка на макет (например, в Figma).

Как генератор выбирает, что показать:

1. Есть `mockup_image` и файл влезает в порог → картинка вшивается в письмо.
   Если при этом задан `mockup_url` — картинка становится кликабельной
   (ведёт на полный макет).
2. Файл слишком большой / не найден / неизвестный формат → картинка НЕ
   вшивается, генератор пишет предупреждение в консоль и показывает
   текстовую ссылку «Макеты» (если задан `mockup_url`).
3. Нет `mockup_image`, только `mockup_url` → просто текстовая ссылка «Макеты».
4. Нет ни того, ни другого → блок макета не показывается.

**Порог размера** — по умолчанию 1536 КБ (1.5 МБ). Настраивается в объекте
`config` в `content.json`:

    "config": { "max_inline_image_kb": 500 }

Поле опциональное: не указали — действует 1.5 МБ.

Почему порог вообще нужен: тяжёлые картинки раздувают письмо, а почтовые
клиенты (особенно Gmail и Outlook) часто режут или не показывают крупные
вшитые изображения. Сжимайте макеты до превью, полноразмер отдавайте через
`mockup_url`.

> Совет: делайте `mockup_image` шириной ~520–1040 px (письмо 600 px шириной,
> учёт retina). Экспорт в сжатый JPG или оптимизированный PNG обычно
> укладывается в порог с запасом.

### Два режима подключения картинок: inline и cid

Поле `image_mode` в объекте `config`:

    "config": { "image_mode": "cid" }

**`"inline"` (по умолчанию)** — картинка кодируется в base64 и вшивается
прямо в HTML (data-URI). Результат — один самодостаточный `email.html`.
Минус: Gmail и десктопный Outlook такие картинки часто не показывают.
Ссылка-фолбэк `mockup_url` подстраховывает.

**`"cid"`** — картинки идут вложениями письма (техника Content-ID):
в HTML вместо base64 стоит ссылка `cid:mockup1@release`, а сам файл
прикладывается к письму как inline-вложение. Это стандартный способ,
который корректно показывают все основные клиенты, включая Gmail и Outlook.

В режиме cid генератор создаёт ТРИ файла:

- `email.html` — HTML со ссылками `cid:`;
- `email.eml` — готовое письмо (MIME multipart/related) для ручной отправки
  через почтовый клиент;
- `email.attachments.json` — манифест вложений для скрипта отправки `send.js`.

### Ручная отправка: пошаговая инструкция (без сервера)

Идея способа: письмо собирается в режиме `inline` — все картинки уже
вшиты внутрь `email.html`. Вы открываете его в браузере и копируете
отрендеренное письмо целиком в окно нового сообщения. Прикладывать
файлы отдельно НЕ нужно — картинки «переезжают» вместе со скопированным
содержимым, и почтовый клиент сам превращает их в свои inline-вложения.

**Шаг 0. Проверьте режим.** В `content.json` в объекте `config` должно
быть `"image_mode": "inline"` (или `config` не указан вообще — inline
действует по умолчанию). Соберите письмо:

    node build.js

**Шаг 1. Откройте `email.html` в браузере.** Двойной клик по файлу или
перетащите его в окно Chrome/Safari/Firefox. Убедитесь, что письмо
выглядит правильно и все картинки на месте.

**Шаг 2. Выделите письмо целиком.** Кликните один раз в любом месте
страницы, затем Ctrl+A (Windows) / Cmd+A (Mac).

**Шаг 3. Скопируйте.** Ctrl+C / Cmd+C.

**Шаг 4. Вставьте в новое письмо.** Дальше зависит от клиента:

*Классический Outlook (Windows):*
1. «Создать сообщение».
2. Убедитесь, что формат письма — HTML: вкладка «Формат текста» →
   выбран «HTML» (не «Обычный текст»).
3. Кликните в тело письма, Ctrl+V.
4. Картинки появятся сразу. Outlook при отправке сам приложит их
   как inline-вложения.

*Outlook в вебе / новый Outlook:*
1. «Создать сообщение».
2. Кликните в тело, Ctrl+V / Cmd+V.
3. Если картинки вставились — готово. Если какая-то не отобразилась,
   см. «Если картинка потерялась» ниже.

*Gmail (веб):*
1. «Написать».
2. Кликните в тело, Ctrl+V / Cmd+V.
3. Gmail вставляет письмо вместе с картинками.

*Apple Mail:*
1. «Новое сообщение».
2. Вставьте Cmd+V. Проверьте, что в настройках Mail формат сообщений —
   «Rich Text» (Настройки → Составление), иначе вставится голый текст.

**Шаг 5. Проверка перед рассылкой (важно!).** Отправьте письмо сначала
самому себе. Откройте полученное и проверьте: картинки на месте, ссылки
кликаются, вёрстка не разъехалась. Только после этого отправляйте команде.

**Если картинка потерялась при вставке.** Некоторые веб-клиенты
пропускают отдельные вшитые картинки. Решение: вставьте картинку руками
прямо в тело письма на её место — в любом клиенте картинку можно
перетащить из папки `mockups/` или вставить через Ctrl+V, предварительно
скопировав сам файл изображения (не путь к нему). Картинка, вставленная
В ТЕЛО письма (не «скрепкой» как вложение), станет inline-вложением
автоматически.

**Чего делать НЕ нужно:**

- НЕ прикладывайте картинки «скрепкой» (как обычные вложения) к письму,
  собранному в режиме `cid`: клиент назначит вложениям свои Content-ID,
  и ссылки `cid:` из HTML с ними не совпадут — в письме будут пустые
  места, а картинки повиснут отдельными файлами.
- НЕ вставляйте письмо в режиме «Обычный текст» — потеряется вся вёрстка.
- НЕ пересылайте `email.html` как вложение-файл: получатели увидят
  просто прикреплённый файл, а не оформленное письмо.

**Известное ограничение:** классический Outlook рендерит вставленный HTML
движком Word — градиент в шапке может упроститься до сплошного зелёного
цвета. Это косметика, остальная вёрстка сохраняется. Проверяется отправкой
самому себе на шаге 5.

### Отправка через SMTP (если доступен сервер)

Самый надёжный способ отправить письмо с картинками — скрипт `send.js`,
который шлёт его напрямую через ваш SMTP-сервер:

    # 1. Один раз: установить зависимости
    npm install

    # 2. Собрать письмо (в content.json должен быть "image_mode": "cid")
    node build.js

    # 3. Проверить без отправки (соберёт preview.eml)
    node send.js --dry-run

    # 4. Отправить
    SMTP_HOST=smtp.company.ru SMTP_FROM=robot@company.ru \
    SMTP_USER=robot@company.ru SMTP_PASS=секрет \
    node send.js --to team@company.ru

Параметры send.js:

| Флаг         | Что делает                                       |
|--------------|--------------------------------------------------|
| `--to`       | Получатели через запятую (обязательно)           |
| `--cc`       | Копия (опционально)                              |
| `--subject`  | Переопределить тему (по умолч. из release_title) |
| `--dry-run`  | Собрать письмо в preview.eml, не отправлять      |

Настройки SMTP — через переменные окружения: `SMTP_HOST` (обязательно),
`SMTP_FROM` (обязательно), `SMTP_PORT` (по умолч. 587), `SMTP_USER`,
`SMTP_PASS`, `SMTP_SECURE=true` для порта 465. Хост и порт вашего
корпоративного SMTP уточните у администраторов почты.

Часть настроек (кроме пароля!) можно положить в `smtp.json` рядом со
скриптом: `{ "host": "...", "port": 587, "user": "...", "from": "..." }`.
Пароль передавайте только через переменную окружения `SMTP_PASS` и не
коммитьте его в репозиторий.

### Ручная отправка .eml (запасной вариант)

В `.eml` добавлен заголовок `X-Unsent: 1` — классический Outlook на Windows
открывает такой файл как неотправленный черновик с кнопкой «Отправить».
Большинство других клиентов (Apple Mail, Thunderbird, новый Outlook)
заголовок игнорируют и открывают письмо как полученное — там остаётся
«Переслать» (убрав «Fwd:» из темы). Если ни то, ни другое не подходит —
используйте отправку через SMTP выше.

Правила порога и фолбэка в обоих режимах одинаковые: файл больше порога →
картинка не добавляется, работает ссылка `mockup_url`.

### Платформы

`platforms` — это массив. Каждый элемент:

    { "type": "ios", "version": "1.84.0" }

| `type`    | Как выглядит бейдж                          |
|-----------|---------------------------------------------|
| `ios`     | зелёный бейдж с логотипом Apple             |
| `android` | зелёный бейдж с логотипом Android           |
| `back`    | голубой бейдж (бэкенд), без логотипа         |

- Порядок и количество — любые.
- `version` можно опустить — тогда покажется только название платформы.
- Указали `type` не из списка — генератор предупредит в консоли и пропустит бейдж.

## Примеры фичей

Картинка макета вшита в письмо + кликабельна (ведёт на полный макет):

    {
      "name": "Новая лента рекомендаций",
      "description": "Гетерогенная лента с курсорной пагинацией.",
      "analytics_url": "https://example.com/analytics/feed",
      "mockup_image": "mockups/feed.png",
      "mockup_url": "https://figma.com/file/feed-full",
      "platforms": [
        { "type": "ios",     "version": "1.84.0" },
        { "type": "android", "version": "1.84.0" },
        { "type": "back",    "version": "3.12.0" }
      ]
    }

Большой файл — не влезет в письмо, останется текстовой ссылкой:

    {
      "name": "Экран настроек VPN",
      "mockup_image": "mockups/settings-big.png",
      "mockup_url": "https://figma.com/file/settings-full",
      "platforms": [
        { "type": "ios", "version": "1.84.0" }
      ]
    }

Только текстовая ссылка на макет, без картинки:

    {
      "name": "Сплит-тоннелинг",
      "description": "Маршрутизация трафика на стороне сервера.",
      "analytics_url": "https://example.com/analytics/split",
      "mockup_url": "https://figma.com/file/split",
      "platforms": [
        { "type": "back", "version": "3.13.0" }
      ]
    }

Голый минимум — только название и платформы:

    {
      "name": "Мелкие правки и стабилизация",
      "platforms": [
        { "type": "ios",     "version": "1.84.0" },
        { "type": "android", "version": "1.84.0" }
      ]
    }

## Частые ошибки в JSON

- Каждый элемент массива (кроме последнего) разделяется запятой; после
  последнего запятой быть НЕ должно. Лишняя запятая — самая частая ошибка.
- Все ключи и текстовые значения — в двойных кавычках: `"name": "..."`.
- Кавычки и угловые скобки внутри текста писать можно — генератор сам их
  экранирует, вёрстка не сломается.
- Ссылки указывайте полностью, с `https://` — иначе в письме они не откроются.

Если JSON оформлен с ошибкой, `node build.js` упадёт с сообщением, в каком
месте файла проблема.

## Заметки по отображению

- Логотипы iOS/Android встроены прямо в HTML (формат SVG), внешние картинки
  не нужны — письмо самодостаточно.
- В десктопном Outlook (рендеринг через движок Word) SVG не отображается:
  там вместо иконки будет просто текст «iOS 1.84.0». Во всех остальных
  клиентах (Gmail, Apple Mail, мобильные приложения, веб-версии) логотип виден.
- Вшитые через data-URI картинки макетов (режим `inline`) в некоторых
  клиентах — прежде всего Gmail и десктопный Outlook — могут не отобразиться.
  Два способа решить: указать `mockup_url` как ссылку-фолбэк, либо
  переключиться на `"image_mode": "cid"` — вложения с Content-ID показывают
  все основные клиенты.
- Папку с картинками (`mockups/`) держите рядом с `content.json`. Пути в
  `mockup_image` указывайте относительно `content.json`.
